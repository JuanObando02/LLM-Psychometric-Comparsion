[SRS-65] El objetivo principal de este documento es describir las necesidades de los usuarios del aplicativo web por medio de historias de usuario al momento de comparar sus rasgos no cognitivos.

Sección

1.2    Alcance

Información

[SRS-66] Este documento especifica las necesidades de los usuarios que accederán al aplicativo web que permite evaluar y comparar rasgos no cognitivos entre modelos LLM y grupos específicos.

Información

[SRS-67] Los usuarios deben poder:

  • Realizar inventarios psicométricos estandarizados.
  • Visualizar resultados de grupos poblacionales y LLM. 
  • Comparar resultados entre LLM y grupo poblacional.
  • Descargar resultados de inventario psicométrico.

1.8    Definiciones

Información

[SRS-85] Big Five: Modelo psicométrico que evalúa cinco grandes rasgos de la personalidad: Apertura, Responsabilidad, Extraversión, Amabilidad y Neuroticismo.

Información

[SRS-86] Dark Triad: Conjunto de tres rasgos de personalidad: Narcisismo, Maquiavelismo y Psicopatía.

Información

[SRS-87] Psicometría: Se encarga de la medición y cuantificación de atributos psicológicos, como la inteligencia, la personalidad o las capacidades cognitivas.

Información

[SRS-88] Pruebas Psicométricas: Son herramientas estandarizadas que miden de forma objetiva y cuantitativa aspectos psicológicos de una persona, como la inteligencia, personalidad, aptitudes y habilidades.

Información

[SRS-89] CSV:  Formato de archivo utilizado para almacenar e intercambiar datos de manera estructurada.

Información

[SRS-90] Interfaz de Usuario (UI): Parte visual del sistema a través de la cual los usuarios interactúan con la aplicación.

Información

[SRS-91] LLM: Es un modelo de inteligencia artificial entrenado con enormes cantidades de texto para entender y generar lenguaje humano de forma natural.

2.   Requerimientos

Sección

2.1    Interfaces Externas

Requisito de interfaz

2.1.1    Interfaz de Usuario (UI)

[SRS-34] El aplicativo web presentará formularios interactivos para contestar inventarios psicométricos, gráficos comparativos y botones para descargar reportes.

Requisito de interfaz

2.1.2    Interfaz de Archivos

[SRS-35] El sistema permitirá exportar resultados en PDF (<5 MB) e importar plantillas en formato Excel con respuestas de grupos poblacionales.

Requisito de interfaz

2.1.3    Compatibilidad

[SRS-37] La aplicación será accesible desde navegadores modernos (Chrome, Firefox, Safari) y dispositivos (PC, tablet, móvil).

Requisito de interfaz

2.1.4    Interfaz con LLM

[SRS-38] Los perfiles de LLM evaluados se cargarán localmente como datos ya procesados, sin necesidad de conexión directa a un modelo externo.

Sección

2.2    Funciones

Requisito funcional

[SRS-33] El sistema debe mostrar las preguntas del inventario psicométrico y permitir que el usuario las responda.

Requisito funcional

[SRS-39] El sistema debe calcular automáticamente los puntajes obtenidos en cada rasgo no cognitivo (Big Five, Dark Triad).

Requisito funcional

[SRS-40] El sistema debe generar un reporte visual de resultados individuales, incluyendo gráficas de barras o radar.

Requisito funcional

[SRS-41] El sistema debe permitir descargar los resultados individuales en formato PDF.

Requisito funcional

[SRS-42] El sistema debe permitir al investigador descargar una plantilla Excel del inventario psicométrico para aplicarla a grupos.

Requisito funcional

[SRS-43] El sistema debe permitir cargar un archivo Excel con respuestas de un grupo poblacional y procesar automáticamente los resultados promedio.

Requisito funcional

[SRS-44] El sistema debe generar un reporte comparativo en el que se visualicen las similitudes y diferencias entre el grupo poblacional y los perfiles de distintos LLM.

Requisito funcional

[SRS-45] El sistema debe permitir descargar el reporte comparativo en formato PDF.

Requisito funcional

[SRS-46] El sistema debe garantizar que los datos procesados se manejen de forma temporal sin almacenamiento en bases de datos.

3.   Verification

Información

[SRS-98] El propósito de esta sección es establecer los métodos, posibles herramientas y criterios mediante los cuales se comprobará que el aplicativo web cumple con los requisitos definidos. La verificación tiene como objetivo proveer evidencia objetiva de que el sistema satisface las especificaciones establecidas.

Sección

3.1    Métodos de verificación y posibles herramientas

Información

[SRS-100]

  • Inspección (documentos, código, interfaces): Se realizarán revisiones de código y de la interfaz para confirmar que cumplen con lo especificado. Para este fin podrían emplearse herramientas como ESLint o Prettier para el control de calidad de código, y Figma para contrastar la interfaz con los prototipos de diseño.

​

Información

[SRS-106]

  • Análisis (cálculos, consistencia, seguridad): Se validará la consistencia de los cálculos psicométricos y la seguridad del sistema en el procesamiento de datos. Entre las herramientas que podrían utilizarse están Postman para pruebas de API y OWASP ZAP para revisiones básicas de seguridad.

​

Información

[SRS-105]

  • Demostración (funcionamiento en entorno de prueba): El sistema se ejecutará en un ambiente controlado para mostrar el flujo completo de uso (aplicación de pruebas, carga de archivos, procesamiento y descarga de reportes). Para esto se podrían emplear contenedores con Docker que permitan aislar el sistema del entorno local.

​

Información

[SRS-104]

  • Pruebas funcionales: Se comprobará que cada requisito se cumple en el sistema. Para estas pruebas podrían utilizarse Cypress para pruebas de extremo a extremo y Jest para pruebas unitarias de funciones críticas como el cálculo de puntajes o la validación de archivos CSV/Excel.

​

Información

[SRS-103]

  • Pruebas no funcionales (rendimiento, compatibilidad): Se evaluará el rendimiento, la accesibilidad y la compatibilidad en navegadores y dispositivos. Herramientas como Lighthouse (incluida en Chrome) o GTmetrix pueden emplearse para estos fines.

​

Información

[SRS-102]

  • Manejo de archivos (CSV/Excel y PDF): Para el procesamiento de resultados y generación de reportes podrían emplearse librerías como SheetJS (xlsx.js) para trabajar con archivos Excel/CSV y jsPDF para generar reportes en PDF.

​

Información

[SRS-101]

  • Privacidad y datos temporales: Se verificarán mecanismos que aseguren que los datos no se almacenen una vez procesados. Para ello se podrían usar Jest con pruebas unitarias específicas y revisiones manuales en el Network tab de Chrome DevTools.

​

Sección

3.2    Criterios de aceptación

[SRS-108]

  • Los inventarios psicométricos se procesan en tiempo real y generan resultados correctos.
  • Los reportes se presentan visualmente y se descargan en PDF con un tamaño ≤ 5MB.
  • El sistema procesa archivos de carga con resultados de al menos 50 participantes.
  • No se almacena información personal ni resultados en ninguna base de datos.
  • El aplicativo es funcional en Chrome, Firefox, Safari y dispositivos móviles, de escritorio y tablets.

​

Sección

3.3    Trazabilidad de requisitos

[SRS-110] Para garantizar que cada requisito funcional y no funcional sea verificado de forma adecuada, se empleará una Matriz de Trazabilidad de Requisitos (RTM). En esta matriz se relacionarán los requisitos del sistema con el método de verificación correspondiente, la herramienta utilizada y el criterio de aceptación. Esto asegurará que no quede ningún requisito sin prueba asociada y permitirá un control riguroso del proceso de verificación.

Sección

3.4    Responsabilidades

[SRS-113]

  • El equipo de desarrollo aplicará inspecciones de código, configurará entornos de prueba y validará la eliminación de datos temporales.
  • El equipo de aseguramiento de calidad (QA) ejecutará pruebas funcionales y no funcionales, documentando incidencias y resultados.
  • El equipo de investigación supervisará la pertinencia de los cuestionarios psicométricos y el cumplimiento de las restricciones éticas y de privacidad.
